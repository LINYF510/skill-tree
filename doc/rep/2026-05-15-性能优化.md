# 性能优化实现报告

## 实现概述

为 Skill-Tree 项目实现全面的性能优化，涵盖 Canvas 渲染层（视口裁剪 + 布局缓存）、数据层（数据库索引 + WAL 模式）、图片加载层（Coil 缓存配置）和调试工具（PerformanceMonitor）。核心目标：节点数超过 200 时帧率不再显著下降。

## 关键代码片段

### 1. 视口裁剪（ViewportCuller）

```kotlin
class ViewportCuller(
    viewportWidth, viewportHeight, scale, offsetX, offsetY, margin = 200f
) {
    val visibleRect: RectF  // 世界坐标系可视区域（逆变换计算）
    fun isNodeVisible(nodePosition, nodeWidth, nodeHeight): Boolean
    fun filterVisibleNodes(positions, nodeWidth, nodeHeight): List<NodePosition>
    fun getVisibleNodeIds(positions, nodeWidth, nodeHeight): Set<String>
}
```

- 200px margin 防止边缘节点闪烁
- 逆变换：`(-offsetX - margin) / scale` 计算世界坐标

### 2. 布局缓存（TreeLayoutCache）

```kotlin
class TreeLayoutCache {
    fun getOrComputeTreeNodes(nodes, compute): List<TreeNode>  // 基于 id+updatedAt hash 缓存
    fun getOrComputePositions(treeNodes, w, h, compute): List<NodePosition>  // 基于画布尺寸缓存
    fun invalidate()  // 节点数据变化时调用
}
```

- `getOrComputeTreeNodes`：通过 `nodes.map { it.id to it.updatedAt }.hashCode()` 检测数据变化
- `getOrComputePositions`：额外检查画布尺寸是否变化
- 使用 `?:` 安全返回避免 `!!`

### 3. SkillTreeScreen 集成

```kotlin
// 缓存实例
val layoutCache = remember { TreeLayoutCache() }
LaunchedEffect(displayNodes) { layoutCache.invalidate() }

// 缓存版 buildTreeNodes
val treeNodes = remember(displayNodes) {
    layoutCache.getOrComputeTreeNodes(displayNodes) { buildTreeNodes(it) }
}

// 缓存版 calculateNodePositions + 视口裁剪
val nodePositions = layoutCache.getOrComputePositions(treeNodes, w, h) { t, w, h ->
    calculateNodePositions(t, w, h)
}
val culler = ViewportCuller(size.width, size.height, scale, offsetX, offsetY)
val visibleNodeIds = culler.filterVisibleNodes(nodePositions, NODE_WIDTH, NODE_HEIGHT)
    .map { it.nodeId }.toSet()

// drawConnections / drawNodes 传入 visibleNodeIds
```

### 4. 动画条件控制

```kotlin
searchPulseAlpha = if (uiState.searchQuery.isNotBlank()) searchPulseAlpha else 0f,
selectedPulseAlpha = if (uiState.selectedNodeId != null) selectedPulseAlpha else 0f,
```

- 动画始终创建（避免 Compose 重组问题），但在绘制时判断是否使用动画值
- 无搜索/无选中时 alpha = 0f，跳过绘制逻辑

### 5. 数据库优化

```kotlin
// 新增索引
indices = [Index("parentId"), Index("nodeType"), Index("title"), Index("createdAt")]

// WAL 模式 + 线程池
.setJournalMode(JournalMode.WRITE_AHEAD_LOGGING)
.setQueryExecutor(Executors.newFixedThreadPool(4))
```

### 6. Coil 缓存配置

```kotlin
ImageLoader.Builder(context)
    .memoryCache { MemoryCache.Builder(context).maxSizePercent(0.15).build() }
    .diskCache { DiskCache.Builder().maxSizeBytes(50L * 1024 * 1024).build() }
```

## 性能优化效果预期

| 优化项 | 预期效果 |
|--------|---------|
| ViewportCuller | 500 节点时仅渲染可见区域（约 20-50 个），绘制量降低 80-90% |
| TreeLayoutCache | 避免每帧重复 buildTreeNodes + calculateNodePositions，CPU 降低 30-50% |
| 动画条件控制 | 无搜索/选中时跳过脉冲动画绘制，减少无效 GPU 工作 |
| 数据库索引 | nodeType/title/createdAt 查询加速 5-10x |
| WAL 模式 | 读写并发性能提升，UI 线程阻塞减少 |
| 查询线程池 | 4 线程并发查询，批量操作延迟降低 |
| Coil 缓存 | 内存限制 15% + 磁盘 50MB，避免 OOM |

## 已知问题

1. `TreeLayoutCache` 使用 `hashCode()` 检测数据变化，理论上存在哈希冲突风险（实际概率极低）
2. `ViewportCuller` 的 `visibleRect` 使用 `lazy` 计算，在同一实例中 scale/offset 变化后不会重新计算——当前每帧创建新实例所以无此问题
3. `PerformanceMonitor.isDebugEnabled` 需在 Application.onCreate 中初始化，若进程重启可能未设置
4. 数据库新增索引使用 `fallbackToDestructiveMigration()`，升级时会清除数据（生产环境需添加迁移）
5. `pointerInput` 中的 `calculateNodePositions` 已替换为缓存版本，但 `layoutCache` 是 Composable 局部变量，pointerInput lambda 捕获的是同一实例
